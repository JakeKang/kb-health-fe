# AI 활용 보고서 (AI_USAGE.md)

이 문서는 KB Health 프론트엔드 과제 제출 과정에서 AI를 어디에 사용했고, 무엇을 사람이 최종 책임지고 확인했는지 현재 제출 상태 기준으로 정리한 보고서입니다.

내용은 `documents/requirement.md`, `documents/openapi.yaml`, 현재 저장소 구현과 문서 상태를 근거로만 작성했습니다.

## 1. 사용한 도구 / 모델

### 1.1 사용한 AI 모델

- OpenAI
  - `gpt-5.4 (plan 검증, 코드 생성, 디자인 구현, 검증 및 테스트 등)`
- Claude
  - `sonnet-4.6 (plan 검증, task 생성, 작업 단위 지정 등)`

### 1.2 작업에 사용한 도구 및 설정

- OpenAI `Codex`
- Anthropic `Claude Code`
- Agent Harness `opencode`
- Agent / Skills 개인 설정
- Read / Glob: `documents/requirement.md`, `documents/openapi.yaml`, `plan/*.md`, `skils/` 등
- 요구사항 문서, openapi 문서, plan 문서, 구현 기준 문서, 문서 검토
- LSP diagnostics: 수정 파일 기준 진단 확인

### 1.3 문서 작성 원칙

- 현재 코드와 문서를 기반으로 분석하여 사실만 기록
- AI가 정리한 범위와 사람이 최종 책임지는 확인 범위를 분리

## 2. 적용한 작업 범위

이번 과제는 전체 방향과 핵심 의사결정을 주도하고, AI를 설계 보조, 개발 보조, 문서화, 검증 보조에 활용하는 방식으로 진행되었습니다.

Plan 및 Task 구성, 폴더 구조, 시스템 아키텍처, 프레임워크·라이브러리 선택, 코드 분할 기준, 디자인 시스템 방향, QA 및 버그 수정 등 핵심 기준을 직접 정하고 구현을 이끌었습니다.

AI는 이 기준을 바탕으로 설계 정리, 구현 보조, 요구사항 대조, 문서 보완, 검증 결과 정리에 활용되었습니다.

### 2.1 설계 영역

- plan / phase / task 작업단위 정리
- 폴더 구조와 코드 분할 방향 반영 정리
- 시스템 아키텍처 및 프레임워크·라이브러리 선택 검증
- shell / auth 흐름, 상태 변화, UX 방향 검토 검증
- 요구사항과 현재 구조의 정합성 점검

### 2.2 개발 영역

- 구현 아이디어 정리 및 페어 형태의 코드 작성 / 보완
- shell / auth follow-up 반영
  - `/sign-in`, global 404의 full-page 구조
  - 401 / refresh 실패 시 `RootLayout` 전역 다이얼로그 후 CTA 이동 흐름
  - guest lock UI, 로그아웃 확인, 로그인 후 `/task` 이동 카드 등 후속 조정
- 로그인 onChange 검증, badge-only status, 상세 중심 status 변경 등 최근 구현 기준 반영
- MSW 더미 데이터 및 구현 의도 정리 보조

### 2.3 문서 영역

- `documents/requirement.md`, `documents/openapi.yaml` 기준으로 현재 구현 재점검
- `README.md`, `AI_USAGE.md`를 현재 구현 기준으로 갱신
- 구현 의도가 중요한 파일(`api/client.ts`, `TaskDetailPage.tsx`, `mocks/handlers.ts`)의 설명 포인트를 문서와 연결

### 2.4 검증 영역

- build 등 검증 명령 실행 및 결과 정리
- e2e 테스트를 통한 task 완료 후 검증 단계 프로세스 진행
- 수정 파일 기준 진단 확인
- 문서와 실제 구현의 정합성 점검
- 다만 최종 해석과 승인 책임은 본인에게 있음

## 3. 핵심 프롬프트 / 지시 방식 요약

실제 작업에서는 반복적으로 필요한 지시를 skills / agent 형태의 기본 틀로 정리해 두고, Claude Code와 Codex에 공통 기준으로 적용했습니다.

### 3.1 설계 관련 지시

- 현재 요구사항, plan, 구현 구조를 먼저 읽고 설계 의도와 충돌하지 않게 정리할 것
- 전체 방향, 구조, 우선순위, 품질 기준은 본인이 직접 결정하고 AI는 그 기준 안에서 보조할 것
- 폴더 구조, 아키텍처, 인증 흐름, 상태 변화, 예외 처리, UX와 같은 핵심 판단은 본인 기준을 우선할 것

### 3.2 개발 관련 지시

- 현재 코드베이스와 폴더 구조를 기준으로 맞는 방향으로만 코드 제안 및 수정할 것
- 단순 생성보다 기존 구조, 구현 의도, 일관성, 유지보수성을 우선할 것
- AI가 제안한 코드나 구조는 그대로 확정하지 않고 검토 가능한 형태로 제안할 것
- 미구현 항목은 구현된 것처럼 작성하지 말 것

### 3.3 문서 관련 지시

- 단계별 구현 사실을 먼저 확인한 뒤 문서화할 것
- 실제 코드와 바로 대조 가능하게 정리할 것
- 요구사항 대응 결과와 차이점은 숨기지 않고 기록할 것
- 확인 가능한 사실만 기록하고, 추정이나 과장은 배제할 것

### 3.4 검증 관련 지시

- build, 진단, 동작 흐름 확인 등 검증 가능한 내용만 정리할 것
- 검증 결과를 AI가 임의로 확정하지 말 것

## 4. 사람이 최종 확인한 내용

### 4.1 설계 확인

- 전체 구조와 설명이 의도한 아키텍처 방향과 일치하는지
- shell / auth follow-up 설명이 현재 라우팅·인증·UX 동작과 맞는지
- AI가 보조한 설계 정리 내용이 실제 코드베이스와 충돌하지 않는지

### 4.2 개발 확인

- 구현 및 수정 내용이 현재 코드베이스와 설계 의도에 부합하는지
- 요구사항 대응 범위와 실제 구현 상태가 과장 없이 반영되었는지
- 미구현 항목이 구현된 것처럼 표현되지 않았는지

### 4.3 문서 확인

- 문서와 실제 구현 사이에 불일치가 없는지
- 요구사항 충족 여부 및 `PARTIAL` 항목 표기가 정확한지
- README와 AI_USAGE가 현재 제출 상태를 기준으로 정리되어 있는지
- 문서와 코드에 민감정보, 비밀값, 개인정보가 포함되지 않았는지

### 4.4 검증 확인

- `pnpm run build` 등 주요 검증 결과가 실제로 유효한지
- 수정 이후 진단 오류나 제출 방해 요소가 남아 있지 않은지
- Playwright E2E 등 미구현 범위가 완료된 것처럼 잘못 기재되지 않았는지

## 5. 정리

이번 과제에서 AI는 설계 보조, 개발 보조, 문서 정리, 요구사항 감사, 검증 보조에 폭넓게 활용되었습니다.  
다만 과제의 핵심 방향 설정, 구조적 판단, 구현 주도, 품질 기준 수립, 최종 제출 책임은 모두 직접 담당했습니다.
